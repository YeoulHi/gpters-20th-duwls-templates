# 10.5.1. 설계 문서 작성 가이드

## 작성 원칙

**원칙 1: 모호함 없음**
- "쉽다" → "단계 3개 이하"
- "복잡하다" → "4단계 이상"

**원칙 2: 누락 없음**
- 기술선택 이유 (3줄 이상)
- 예상 실패 지점 (최소 3개)
- 제약사항별 해결방안

**원칙 3: 검증 가능**
- 개발팀이 읽고 바로 구현 가능 수준
- 불명확한 표현 금지

---

## 각 섹션별 작성 방법

### 1️⃣ 기술 스택 결정

**작성 수준: 근거 포함 상세화**

❌ 나쁜 예:
\`\`\`
기술: Python, Selenium
선택 이유: 좋아서
\`\`\`

✅ 좋은 예:
\`\`\`
기술: Python 3.11.7, Selenium 4.15
선택 이유:
- 웹 크롤링 필요 → Selenium 표준 선택
- 기존 프로젝트가 Python 3.11.7 → 버전 통일
- Windows/Linux 모두 지원 필요 → 크로스플랫폼 도구

대체 기술 검토:
- Playwright: 더 빠르지만, 팀 경험 없음
- Pupeteer: Node.js 필요 → Python 프로젝트와 이질적
\`\`\`

---

### 2️⃣ 아키텍처 설계

**작성 수준: 단계별 + 데이터 흐름 명확**

❌ 나쁜 예:
\`\`\`
프로세스: 입력 → 처리 → 출력
\`\`\`

✅ 좋은 예:
\`\`\`
Phase 1: 데이터 수집
- 입력: CSV 파일 (경로: ~/data/input.csv)
- 처리: pandas.read_csv() → 데이터 검증
- 출력: 검증된 DataFrame (메모리)
- 예상 시간: 2초 (100MB 파일 기준)

Phase 2: 데이터 변환
- 입력: DataFrame
- 처리: 컬럼 매핑 → 중복 제거 → 정렬
- 출력: 정제된 DataFrame
- 예상 시간: 5초

Phase 3: 결과 저장
- 입력: DataFrame
- 처리: 형식 변환 → 파일 저장
- 출력: output.xlsx (경로: ~/results/)
- 예상 시간: 1초

전체 소요 시간: 8초 (기존 15분 대비 99.9% 단축)
\`\`\`

## 📋 설계 문서 최종 체크리스트

작성 완료 전 확인:

- [ ] 기술선택 근거가 3줄 이상인가?
- [ ] 아키텍처가 Phase별로 명확한가?
- [ ] 에러 처리 지점이 3개 이상 있는가?
- [ ] 각 에러별 처리 방식이 명시되어 있는가?
- [ ] 의존성에 버전이 명시되어 있는가?
- [ ] 성공 지표가 측정 가능한 숫자인가?
- [ ] 개발팀이 읽고 바로 구현할 수 있는 수준인가?
